# AI-Native 知识库方法论

> **一句话定位**：让 AI 真正嵌入你的知识型工作流，不是"用 AI 写一次",是"AI 持续参与一项工作"。
>
> **核心信念**：知识型工作的终极产物不是结论文档,是"可被 AI 和人共同协作演进"的知识库。

---

## 零、AI 交互原则（本 skill 的行为准则）

> **这一节 AI 必须严格遵守，它决定了用户体验的自然度。**

1. **第一句永远问项目，不介绍方法论**
   ✅ "这个项目是关于什么的？"
   ❌ "我来帮你按 ai-native-knowledge-base 方法论建知识库"

2. **Level 判别完全内化**
   用户不需要知道什么是 L1/L2/L3/L4。AI 自己判，需要升级时才委婉提示。

3. **开工前只问 2-3 个问题**
   不要问卷式轰炸。信息不够可以边做边问。

4. **主动报告影响面**
   改动前说清楚"会影响什么"，不要等用户问。

5. **治理动作包装为价值描述**
   ✅ "这个改动影响下游数据，我起草一份影响评估你确认"
   ❌ "根据 L4 治理规则，必须产出 ADR 和影响评估"

6. **术语少用，白话优先**
   SSoT / ADR / changelog / 联动检查 —— 这些词用户能懂最好，听不懂 AI 主动换成人话。

7. **机制是副产品，不是目的**
   用户想做事,机制跟着事自然产生。不是"为了遵守机制而做事"。

## 一、什么时候用本 skill

### ✅ 适合

- 任何**复杂度较高**的知识型工作（概念多 / 规则多 / 要长期迭代）
- 希望 AI **持续**参与（而不是每次对话从零解释）
- 改错了要能**追溯**原因（三个月后还记得"为什么这么定"）
- 改一处可能**影响**其他内容或下游（需要联动检查）

### ❌ 不适合（ANTI-PATTERNS）

- **一次性任务**：一封邮件、一个答案、一段代码 → 直接用 AI 对话即可
- **纯代码仓库**：GitHub README + 代码组织就够 → 用软件工程的成熟框架（ADR / C4 model）
- **个人流水账笔记**：每日日记、碎片感想 → 用 Obsidian / Notion 的双向链接即可
- **已有成熟框架的领域**：学术期刊有官方 LaTeX 模板 / 监管合规有法定格式 → 用行业标准

## 二、核心元模型（元三问）

任何知识型工作本质都在回答三个问题：

| 问题 | 对应目录 | 必选？ |
|---|---|---|
| **关于什么？**（What）| `00-charter/` 宪章层 | ✅ 必选 |
| **怎么组织？**（How to structure）| `10-<domain>/` 领域层 | ✅ 必选 |
| **怎么演进？**（How to evolve）| `40-process/` 过程层 | ✅ 必选 |

其他层都是**可选工具**，按需启用：

| 层 | 什么时候启用 |
|---|---|
| `20-specs/` 规范层 | 有共识规范 / 规则 / 守则需要沉淀 |
| `30-resources/` 资源层 | 有结构化数据（CSV/JSON）、文献库（BibTeX）、设计 tokens 等 |
| `99-raw/` 原始层 | 有他人给的原始材料需要只读保存 |

**3 层必选 + 3 层可选**。不是每个项目都要上 6 层。

## 三、Level 梯度（AI 嵌入深度）

**本质**：Level 不是档次,是"你希望 AI 承担多少职责 + 项目能容忍多大错误"。

### L1 · Assist（辅写）

- **AI 角色**：打字帮手。你写主体,AI 补空、润色、查资料
- **场景**：单次任务 / 内容偏线性 / 没有规则要记
- **时间跨度**：1-2 周内
- **目录需要**：`CLAUDE.md` + 1 份主文档（仅此而已）
- **治理机制**：无

### L2 · Remember（留痕）

- **AI 角色**：记忆助手。AI 持续记住项目背景、核心概念、历史决定
- **场景**：有脉络,但规则仍在形成 / 需要回溯"上次决定过什么"
- **时间跨度**：1-3 个月
- **目录需要**：+ `00-charter/`、`40-process/inbox/`
- **治理机制**：inbox 分拣 + 概念词表维护

### L3 · Operate（运转）

- **AI 角色**：共工。AI 基于规则主动执行,你做方向性决策
- **场景**：有成型规则 / 反复使用 / 改一处牵连多处
- **时间跨度**：3 个月以上
- **目录需要**：完整 6 层（视情况启用 specs/resources/raw）
- **治理机制**：ADR + changelog + 联动检查

### L4 · Guard（守护）

- **AI 角色**：守护者。AI 拦截风险、强制留痕、逼出影响评估
- **场景**：改动影响**下游**（数据 / 模型 / 他人工作 / 外部用户）→ **回退不起**
- **时间跨度**：长期维护
- **目录需要**：L3 所有 + 影响评估模板
- **治理机制**：L3 所有 + 影响评估强制 + Owner Accept 变更审批

### L3 vs L4 的区别（关键）

**用一个具体场景看**：改一条标签定义

| 动作 | L3 做法 | L4 做法 |
|---|---|---|
| 提出改动 | AI 直接起草 | AI 先问：为什么改 / 影响哪些场景 / 下游会感知吗 |
| 决策留档 | 建 ADR | ADR + **必填**影响评估（量化漂移） |
| 落地 | 直接改 | 必须 Owner 明确 Accept 才改 |
| 联动检查 | 扫一遍 | 扫 + **生成检查报告存档** |
| 下游通知 | 口头 | 生成正式对齐文档 |

**一句话**：L3 AI 执行 + 你决策；L4 AI 把关 + 你走流程才能改。

## 四、Level 自动判别（AI 在对话中内化的判别逻辑）

> **用户不需要知道有这个判别**。AI 听 2-3 轮对话就心里有数。

### 四个观察维度

| 维度 | 观察信号 |
|---|---|
| **A. 复杂度** | 涉及多少独立的概念/规则/决策？简单 → L1-L2；十几条以上 → L3-L4 |
| **B. 时间跨度** | 1-2 周 → L1；1-3 月 → L2；3 月+ → L3；长期 → L4 |
| **C. 可追溯性** | 三个月后需要知道"为什么当初这么定"？→ 至少 L3 |
| **D. 下游影响** | 改动影响外部（数据/模型/他人）？→ L4 |

### 快速判别口诀

```
if 改错了有下游受害者 OR 改动影响评分/决策
    → L4
elif 有成型规则 AND 会长期迭代
    → L3
elif 有项目脉络 AND 需要 AI 持续记住
    → L2
else
    → L1
```

### Level 升级触发信号

AI 在日常运行中观察到以下信号时,**主动建议**升级（不擅自升）：

| 当前 Level | 升级信号 |
|---|---|
| L1 → L2 | 出现"上次好像定过但忘了" / AI 每次重新问背景 |
| L2 → L3 | 同一类规则被引用 3+ 次 / 开始有下游使用 / inbox 堆积 10+ 条 |
| L3 → L4 | 改动开始影响外部 / 出现"改 A 忘同步 B" / 决策正确性需要验证 |

## 五、标准目录结构

```
<project>/
├── CLAUDE.md                   # 给 AI 的协作契约（必读）
├── README.md                   # 给人的入口
│
├── 00-charter/                 # 宪章层 · 必选
│   ├── 01-项目背景与目标.md      # Why / What / 不做什么
│   └── 02-核心概念词表.md         # 术语定义（SSoT）
│
├── 10-<domain>/                # 领域核心层 · 必选
│   ├── 00-总则.md               # 该领域的总体框架
│   ├── 10-*.md / 11-*.md        # 按子维度 / 子模块拆
│   └── 20-<subscene>/           # 按场景分型（如需）
│
├── 20-specs/                   # 规范层 · 可选（有规则/规范时启用）
│   ├── 00-总览.md
│   ├── 10-操作手册.md
│   └── 20-专项规则.md
│
├── 30-resources/               # 资源层 · 可选（有结构化数据/文献/tokens 时启用）
│   ├── 00-架构说明.md
│   ├── 10-<data>.csv / .bib / .json
│   └── 11-<data>-changelog.md
│
├── 40-process/                 # 过程层 · 必选
│   ├── inbox/                   # 唯一输入入口
│   ├── decisions/               # ADR-0001 / 0002 ...
│   └── rule-history.md          # 外部规则变更归档
│
└── 99-raw/                     # 原始层 · 可选（有原始材料时启用）
    └── *.docx / *.mp3 / *.pdf
```

**命名规则**：
- 数字前缀控制显示顺序（`00-` > `10-` > `20-`）
- 同层文件用双位数（`10-` / `11-`）留扩展空间
- 中文无碍,**避免空格**（用连字符或全中文）

## 六、两种启动模式

### 6.1 Greenfield · 新建项目

**触发语**："我要做一个 XX 项目" / "怎么开始写这个论文" / "从零搭一个知识库"

**标准流程**（AI 主导,用户最多回答 3 问）：

```
Step 1. 问项目信息（2-3 问）
   - 这是关于什么的？
   - 预计持续多久？
   - 有现成的材料吗？

Step 2. AI 内化判别
   - 识别领域 → 自动加载对应领域包（见 §七）
   - 判别 Level → 决定启用多少层 + 多少机制

Step 3. 创建骨架
   - 生成目录结构
   - 填充 CLAUDE.md（按领域定制）
   - 起草 00-charter/ 两份文档
   - 创建 ADR-0001（采用本方法论的决策）

Step 4. 给用户"下一步 Checklist"
   - 你需要提供哪些原始材料
   - 你需要定义哪些维度
   - 第一个可落地的产出是什么
```

**示例完整对话**：见 `references/demos/greenfield-论文.md`

### 6.2 Retrofit · 改造已有项目

**触发语**："帮我整理这个文件夹" / "文档结构乱了" / "[拖入已有路径]"

**标准流程**（三步,分批执行,零风险先做）：

```
Step 1. 诊断扫描
   - 递归扫描目标文件夹
   - 对每个文件分类：
     🟢 明确是原始材料（→ 99-raw/）
     🟡 已加工结论（→ 10-<domain>/ 或 20-specs/）
     🟠 疑似草稿/中间版/重复版（需用户判断）
     🔴 难判定（需用户告知）
   - 识别已有"隐性 Level"
   - 推荐目标 Level

Step 2. 出方案
   - 画"当前 vs 重构后"对比图
   - 列迁移清单（哪个文件移到哪）
   - 标红需要用户决策的风险项
   - 给预估时间

Step 3. 分批执行
   批次 1（零风险）：AI 直接做
     - 原始材料统一进 99-raw/
     - 生成新骨架文件（CLAUDE.md、charter 等）
   批次 2（需确认）：用户一条条 Accept
     - 重复版本取舍
     - 内容归类有歧义的
   批次 3（兜底）：
     - 输出"重构前后对照"总结
     - 保留原始文件的"废弃"标记（不真删）
```

**重要约束**：
- **永远不真删文件**,最多标"废弃"或移到 `.archive/`
- **保留原始结构快照**（git 提交 / zip 备份）以便回滚
- **迁移 = 移动**,不是删除

**示例完整对话**：见 `references/demos/retrofit-混乱文件夹.md`

## 八、日常运转的 4 条核心操作

### 8.1 收到新输入

```
用户扔来 会议纪要 / 群消息 / 文档 / 灵感
  ↓
AI 判断类型（不问用户,自己判）:
  ├─ 原始 → 存进 inbox/ 或 99-raw/
  └─ 已加工 → 更新到对应层文件
  ↓
AI 提取"可行动项" + "需共识的分歧"
  ↓
用户确认 → 落地 → 触发联动检查
```

### 8.2 做方向性决策

```
识别到是方向性改动
  ↓
(L3+) 起草 ADR（背景 / 选项 / 决策 / 影响）
(L4)  + 影响评估（量化漂移）
  ↓
用户 Accept
  ↓
落地 + 联动检查 + 更新派生物
```

### 8.3 改 SSoT 数据

```
改 CSV / JSON / 核心词表
  ↓
追加 changelog 条目
  ↓
全局搜索所有引用
  ↓
活引用必改 / 历史陈述保留
  ↓
派生物重新生成（Excel / 图 / HTML）
```

### 8.4 联动检查（§6.4 核心机制）

每次改 SSoT 后,AI **必须主动扫**并报告：

| 检查项 | 动作 |
|---|---|
| 全局搜旧名称 | `grep -r "旧名"` |
| 分类匹配 | 活引用（必改） vs 历史陈述（ADR/changelog 保留） |
| 派生物同步 | Excel / 图 / HTML 需重新生成？ |
| CLAUDE.md 更新 | 必读清单是否需调整？ |
| 决策留档 | 需要 ADR / changelog？ |

详见 `references/cross-ref-check.md`。

## 九、常见陷阱与对策

| 陷阱 | 征兆 | 对策 |
|---|---|---|
| **SSoT 多处复制** | 同一规则出现在 3 份文档里 | 只在一处权威定义,其余用链接引用 |
| **原始派生混存** | docx 和加工稿放一起 | 分层：`99-raw/` 只读,`10-*/` 派生 |
| **决策无留档** | 三个月后没人记得为什么这么定 | 方向性决策必 ADR（L3+） |
| **改 A 忘改 B** | 用户发现某处还是旧名 | 改动后强制联动检查 |
| **Inbox 永远满** | 输入积累不分拣 | 每次有新输入,AI 主动分拣 |
| **CLAUDE.md 太长** | 必读清单 20 份 AI 读不完 | 控制在 3-5 份,其他按需查阅 |
| **维度错切** | Step 3 切完发现某层子项全空 | 切之前和用户深聊,确认切面正交 |
| **治理过度** | 小项目上 L4,维护成本爆炸 | 按 Level 判别,不盲目上严格机制 |
