# 领域包 · 产品规范 / PRD

## 推荐 Level

L2-L3。需求频繁变 + 有跨团队依赖 → L4。

## 启动问用户

1. 产品是 B 端还是 C 端？
2. 是新产品从 0 到 1,还是老产品迭代？
3. 谁会用这份 PRD？（只自己 / 和开发对齐 / 过 Review 委员会）

## 定制化目录

```
10-specs/
  ├── 00-产品总体定位.md       Why / What / 不做什么
  ├── 10-用户画像.md
  ├── 20-核心流程.md           用户旅程
  ├── 30-功能模块/
  │   ├── 00-索引.md
  │   ├── 10-<模块1>.md
  │   └── 11-<模块2>.md
  └── 90-examples/              UI / 交互示例

20-specs/                     规范
  ├── 00-优先级矩阵.md
  └── 10-决策框架.md            P0/P1/P2 判定

30-resources/
  ├── 10-requirements.csv       需求池（SSoT）
  ├── 11-requirements-changelog.md
  └── designs/                   设计稿链接
```

## 常见输入分拣

| 输入 | 处置 |
|---|---|
| 用户反馈 / 调研数据 | inbox/ → 汇总后进 `需求池` |
| 竞品分析 | inbox/ → 可能成为新功能点 |
| 开发评审反馈 | inbox/ → 可能触发需求调整 |
| 数据分析结果 | 99-raw/data/ |
| 功能增删提议 | 建 ADR |

## 典型 ADR

- ADR-0001：产品核心定位确定
- ADR-0002：某功能做 vs 不做的决策
- ADR-0003：优先级策略调整
- ADR-0004：技术方案选型影响产品

## CLAUDE.md 片段

```markdown
## 项目专属约定

| 类型 | 处置 |

| 竞品信息 | inbox/competitive/ |
| 评审意见 | inbox/review/ |

```
