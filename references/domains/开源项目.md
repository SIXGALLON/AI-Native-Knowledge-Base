# 领域包 · 开源项目文档

## 推荐 Level

L3。开源项目天生需要 ADR（软工本就有这个传统）+ 完整文档体系。

## 启动问用户

1. 这个开源项目做什么的？（library / tool / framework / skill）
2. 目标用户是开发者还是终端用户？
3. 现在什么状态？（待发布 / 已有用户 / 维护期）

## 定制化目录

```
README.md                     GitHub 首页门面
LICENSE
CONTRIBUTING.md
CODE_OF_CONDUCT.md

10-docs/
  ├── 00-overview.md
  ├── 10-getting-started.md
  ├── 20-guides/                使用指南
  │   ├── 01-<场景1>.md
  │   └── 02-<场景2>.md
  ├── 30-api/                   API 参考
  ├── 40-advanced/              高级用法
  └── 90-examples/               示例项目

20-specs/
  ├── 00-代码规范.md
  ├── 10-API 设计原则.md
  └── 20-发布流程.md

30-resources/
  └── changelog.md               按版本记录变更

40-process/
  ├── inbox/
  │   ├── issues/                 GitHub issue 归档
  │   └── discussions/
  └── decisions/                  ADR（技术决策）

99-raw/                        （通常不需要）
```

## 常见输入分拣

| 输入 | 处置 |
|---|---|
| GitHub issue | inbox/issues/ → 分类为 bug / feature / question |
| Pull Request 讨论 | inbox/discussions/ |
| 用户反馈 / 邮件 | inbox/feedback/ |
| 架构技术选型 | 建 ADR |
| Breaking change | 建 ADR（必含 migration guide） |

## 典型 ADR

- ADR-0001：技术栈选型
- ADR-0002：API 设计范式
- ADR-0003：版本发布策略（SemVer / CalVer）
- ADR-0004：某 breaking change 的决策
- ADR-0005：deprecation 策略

## 特有机制

### 版本与 CHANGELOG

严格遵循 [Keep a Changelog](https://keepachangelog.com/) 格式：

```
## [Unreleased]
## [1.2.0] - 2026-04-24
### Added
### Changed
### Deprecated
### Removed
### Fixed
### Security
```

### 贡献者协作

`CONTRIBUTING.md` 必须包含：
- 开发环境搭建
- 提交 PR 的流程
- Code style
- Review 标准

## CLAUDE.md 片段

```markdown
## 项目专属约定

- GitHub 相关工作流走 issues / PR,重大决策沉淀到 ADR
- CHANGELOG.md 严格遵循 Keep a Changelog 格式
- Breaking change 必须 ADR + migration guide + 提前 deprecation notice

## 输入分拣

| 类型 | 处置 |
|---|---|
| issue | inbox/issues/ |
| PR 讨论 | inbox/discussions/ |
| 技术决策 | 建 ADR |
| 版本发布 | 改 CHANGELOG.md |
```
