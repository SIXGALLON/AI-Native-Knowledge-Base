# 领域包 · 法律合规 / 政策

## 推荐 Level

**L4 必选**。合规文档改错了有法律后果,天生需要严格治理。

## 启动问用户

1. 涉及哪些监管框架？（GDPR / CCPA / 网安法 / 数据保护法 / 行业规范）
2. 内部政策还是对外合规声明？
3. 是持续维护还是一次性编制？

## 定制化目录

```
10-policies/
  ├── 00-总则.md                合规整体框架
  ├── 10-<领域1>/                如：数据保护 / 反垄断 / 反欺诈
  │   ├── 00-政策.md
  │   └── 10-实施细则.md
  ├── 20-domains/                按监管领域分类
  └── 90-cases/                   判例 / 先例
      ├── compliance/              合规案例
      └── violations/              违规案例

20-specs/
  ├── 00-监管框架清单.md
  └── 10-审计清单.md

30-resources/
  ├── 10-clauses.csv              条款库（SSoT）
  ├── 11-clauses-changelog.md
  ├── regulations/                 法规原文
  └── legal-opinions/              律师意见

40-process/
  ├── inbox/
  ├── decisions/                   ADR（每次政策变更必有）
  └── impact-assessments/           影响评估（L4 强制）
```

## 常见输入分拣

| 输入 | 处置 |
|---|---|
| 新法规发布 | `99-raw/regulations/` + 触发政策审查 |
| 律师意见 / 法律咨询 | `30-resources/legal-opinions/` |
| 违规事件 | inbox/violations/ → 可能更新政策 |
| 监管部门反馈 | inbox/regulator/ → 优先级最高 |
| 政策修订 | 建 ADR + 影响评估 + 法务审核 |

## 典型 ADR

- ADR-0001：合规体系建立
- ADR-0002：某政策修订
- ADR-0003：适应新法规（如 GDPR 生效）
- ADR-0004：跨境数据处理策略

## 特有机制

### 法律版本追溯

每条政策必须记录：
- 依据的法规版本
- 生效日期
- 过期日期（如有）
- 替代的旧版本

### 影响评估强制

任何政策修订的 ADR 必须包含：
- 影响的业务范围
- 对用户/合作伙伴的影响
- 过渡方案
- 内部培训计划
