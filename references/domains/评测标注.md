# 领域包 · 评测 / 标注

> 任何评测项目（模型评测、RM 训练数据标注、质量打标等）的 quick-start。

## 推荐 Level

**通常是 L3-L4**：
- L3 如果只是单人做一次性评测任务
- L4 如果会影响 RM 训练 / 下游数据 pipeline（回退不起）

## 启动时问用户的 3 个问题

1. 评测的对象是什么？（大模型输出 / 产品功能 / 人工回答）
2. 有多少个评估维度 / 标签？（决定复杂度）
3. 打完的分 / 标要给谁用？（决定下游影响 → Level）

## 定制化目录

```

  ├── 00-总则.md              理想态整体框架
  ├── 10-<维度1>.md           如：信息密度 / 事实准确 / 对话感
  ├── 11-<维度2>.md
  ├── 20-场景分型/             按任务类型
  │   ├── README.md
  │   ├── R1-结论型.md
  │   └── R2-解释型.md ...
  └── 90-案例库/
      ├── good/               正例
      └── bad/                反例

20-eval-rules/               规范（打分规则）
  ├── 00-评测规则总览.md       DCG / P 级 / 权重（SSoT）
  ├── 10-标注操作手册.md       SOP
  └── 20-专属规则.md           子场景规则

30-resources/                资源
  ├── 10-labels.csv            标签字典（SSoT）

  ├── 11-labels-changelog.md   标签变更日志
  └── 00-标签体系架构.md
```

## 常见输入分拣

| 输入类型 | 处置 |
|---|---|
| 算法/产品给的评测需求文档 | `99-raw/` |
| 标注员疑问 / 边界 case | `40-process/inbox/YYYY-MM-DD-issue-N.md` |
| 对齐会议纪要 | `40-process/inbox/YYYY-MM-DD-meeting-主题.md` |
| 标签新增/合并/拆分提议 | 建 ADR + 改 CSV + 改 changelog |
| 打分规则调整 | 建 ADR（必含影响评估）+ 改规则总览 |

## 典型 ADR 主题

- ADR-0001：采用知识库体系
- ADR-XXXX：某组标签合并 / 拆分
- ADR-XXXX：打分规则调整（维度权重 / 阈值 / GSB 判定）
- ADR-XXXX：P 级梯度调整
- ADR-XXXX：存量数据回溯策略

## 特有机制

### 打分影响评估（L4 必备）

改任何影响分数的东西,ADR 里必须有：

```markdown
## 打分影响评估

| 场景 | 旧分数 | 新分数 | 差异 |
|---|---|---|---|
| ... | ... | ... | ... |

### 存量数据处理
- <历史打标的回溯策略>
```

### 联动检查特殊关注点

改了标签定义 / 标签 CSV → 必扫：
- 所有理想态文档（正反例里的标签引用）
- 标注操作手册的判重规则
- 派生 Excel 文件重新生成
- ADR 里的引用（保留历史陈述）

## CLAUDE.md 片段（可直接粘贴）

```markdown
## 项目专属约定

- `30-resources/10-labels.csv` 是标签字典的唯一真源,改了必同步 changelog + xlsx
- 打分规则以 `20-eval-rules/00-评测规则总览.md §三` 为准
- 存量打标数据不改,除非 ADR 明确要求回溯
- `99-raw/` 下文件永远不编辑

## 输入分拣

| 输入类型 | 处置 |
|---|---|
| 评测需求文档 / 标注规范原稿 | 存 99-raw/ |
| 标注员疑问 | inbox/YYYY-MM-DD-issue-N.md |
| 算法/产品对齐会 | inbox/YYYY-MM-DD-meeting-主题.md |
| 标签或规则变更提议 | 建 ADR,不直接改 |
```
