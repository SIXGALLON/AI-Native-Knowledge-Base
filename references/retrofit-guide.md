# Retrofit 模式 · 改造已有项目的完整指引

> 用户拖入已有文件夹或说"帮我整理一下"时,AI 按本流程操作。

---

## 核心约束（红线,不可违反）

1. **永远不真删文件**,最多移到 `.archive/` 或标"废弃"
2. **先诊断再动手**,不能上来就改
3. **分批执行**,零风险的先做,有风险的必须 Owner 确认
4. **输出对照**让用户清楚看到"改了什么"
5. **保留快照**以便回滚（git commit 或 zip）

---

## 三步标准流程

### Step 1 · 诊断扫描

**AI 动作**：

1. 递归列出所有文件（含深层子目录）
2. 对每个文件判类型：

   | 类型 | 判断依据 | 处置方向 |
   |---|---|---|
   | 🟢 **原始材料** | `.docx` `.pdf` `.mp3` `.pptx` `.xlsx` 源文件 / 明显是他人给的 | → `99-raw/` |
   | 🟡 **已加工结论** | 正式 markdown / 清晰标题结构 / 自己写的 | → `10-<domain>/` 或 `20-specs/` |
   | 🟠 **疑似草稿** | 文件名含 `draft/草稿/v1/v2/旧/temp` / 最后修改久远 | 需用户判断 |
   | 🟠 **重复版本** | `方案.md` `方案v2.md` `方案final.md` 同主题多份 | 需用户选留哪份 |
   | 🔴 **难判定** | 文件名模糊 / 内容少难判 | 询问用户 |

3. 识别已有的"隐性 Level"：

   - 平铺无目录 → 隐性 L1
   - 有简单分类 → 隐性 L2
   - 有规则文档但散落 → 隐性 L3 但未落实
   - 有下游依赖但无 ADR → **需要 L4 但裸奔**

4. 推荐目标 Level（按 SKILL.md §三 的判别逻辑）

**输出**：诊断报告

```markdown
## 扫描结果

📂 `<path>`
总计 N 个文件,M 个目录

### 分类

🟢 明确原始材料（X 个）：
  - file1.docx
  - file2.pdf
  ...

🟡 已加工结论（Y 个）：
  - ...

🟠 疑似草稿或中间版（Z 个）：
  - ...

🔴 难判定（W 个）：
  - ???.md（内容只有 3 行）
  - 新建 markdown.md（文件名是默认值）
  ...

### Level 判断

现在像是：隐性 L2,但有若干 L3 特征没落实
推荐目标：L3 Operate
理由：
  - 你提到 <规则改动频繁 / 反复用>
  - 文档数量 <N> 份,需要体系化管理

### 下一步

我先给你出迁移方案,你过目 OK 了再动手。
```

---

### Step 2 · 出方案

**AI 动作**：

1. 画"当前 vs 重构后"对比图：

```
当前                            重构后
<path>/                         <path>/
├── 方案v1.md                   ├── CLAUDE.md ✨新增
├── 方案v2.md                   ├── README.md ✨新增
├── 方案final.md                ├── 00-charter/
├── 会议 0415.docx              │   ├── 01-项目背景与目标.md ✨
├── notes.md                    │   └── 02-核心概念词表.md ✨
├── 规则.md                     ├── 10-<domain>/
├── data.xlsx                   │   └── 00-总则.md（← 由 方案final.md 改名）
└── raw/                        ├── 20-specs/
    └── 客户需求.pdf            │   └── 00-规则总览.md（← 由 规则.md 改名）
                                ├── 30-resources/
                                │   └── 10-data.xlsx
                                ├── 40-process/
                                │   ├── inbox/
                                │   │   └── 2025-04-15-会议.docx（← 由 会议 0415.docx 改名）
                                │   └── decisions/
                                │       └── ADR-0001-采用知识库体系.md ✨
                                └── 99-raw/
                                    └── 客户需求.pdf（← 由 raw/ 移入）

.archive/ ✨新增（废弃版本存放）
├── 方案v1.md
└── 方案v2.md（被 方案final.md 代替）
```

2. 列迁移清单（表格）：

| 当前位置 | 目标位置 | 风险 | 是否需确认 |
|---|---|---|---|
| `raw/客户需求.pdf` | `99-raw/客户需求.pdf` | 低 | ❌ AI 直接做 |
| `会议 0415.docx` | `40-process/inbox/2025-04-15-会议.docx` | 低 | ❌ AI 直接做 |
| `data.xlsx` | `30-resources/10-data.xlsx` | 低 | ❌ AI 直接做 |
| `规则.md` | `20-specs/00-规则总览.md` | 中 | ✅ 需确认（重命名对不对） |
| `方案final.md` | `10-<domain>/00-总则.md` | 中 | ✅ 需确认（内容归类对不对） |
| `方案v1.md`, `方案v2.md` | `.archive/` | 中 | ✅ 需确认（确实可以废弃吗） |
| `notes.md` | 不明 | 高 | 🔴 需用户说明 |

3. 标红需要用户决策的：

```markdown
⚠️ 需要你决策的 N 项：

1. `方案v1.md` `方案v2.md` 确实废弃吗,还是有内容需要合并进 final？
2. `notes.md` 是谁写的笔记,要保留吗？放哪？
3. `规则.md` 应该归到 `20-specs/` 总览还是拆成多个子规则？
```

4. 给预估时间：

```
Step 3 执行预估：
  批次 1（零风险,AI 做）: 3 分钟
  批次 2（需你确认）: 10-15 分钟（看你决策速度）
  总计约 15-20 分钟
```

---

### Step 3 · 分批执行

#### 批次 1 · 零风险操作（AI 直接做）

- 备份：`git init && git add . && git commit -m "Snapshot before restructure"`（或创建 zip）
- 创建新目录骨架
- 生成 CLAUDE.md、Charter 骨架、ADR-0001
- 移动明确原始材料到 `99-raw/`
- 移动明确会议纪要到 `inbox/`
- 移动数据文件到 `30-resources/`

**完成后报告**：

```
✅ 批次 1 完成（耗时 2 分钟）
- 快照已保存（git commit <hash>）
- 新目录已创建
- 12 个零风险文件已移到位
- CLAUDE.md 已初始化（Level = L3）
- ADR-0001 已起草

现在进入批次 2,需要你逐项确认:
```

#### 批次 2 · 需用户确认（逐项）

对每个高风险项,AI 问：

```
❓ `方案v1.md` vs `方案v2.md` vs `方案final.md`

内容速读：
- v1: 关于 ABC 的初稿（3000 字）
- v2: 在 v1 基础上加了 XYZ 章节（5000 字）
- final: 重写了,删掉了部分 v2 内容（4200 字）

建议：
  - final 保留为主文档
  - v1 移到 .archive/（简单初稿,无独特价值）
  - v2 移到 .archive/,但提醒你 XYZ 章节没进 final,确认是否故意

你怎么处理？
  A. 按建议执行
  B. 保留 v2 的 XYZ 章节,合并进 final
  C. 都保留,我自己决定
```

**用户回答后,AI 执行该条**。

#### 批次 3 · 兜底清理 + 对照总结

所有迁移完成后：

1. 二次扫描,确认目标结构完整
2. 生成"重构前后对照"总结文档：

```markdown
# 重构总结 · YYYY-MM-DD

## 数字对比

| 指标 | 重构前 | 重构后 |
|---|---|---|
| 顶层文件 | 18 | 3 |
| 目录层级 | 平铺 1 层 | 结构化 6 层 |
| 重复版本 | 有（v1/v2/final）| 已清理 |
| 原始 vs 派生 | 混存 | 分层 |
| Level | 隐性 L2 混乱 | L3 Operate |

## 主要迁移

[完整迁移清单]

## 废弃到 .archive/ 的文件（可随时恢复）

[列表]

## 快照

git commit <hash>（如需回滚）
zip 备份: backup-YYYY-MM-DD.zip

## 下一步建议

1. 阅读新生成的 CLAUDE.md,确认项目定义
2. 填充 00-charter/ 两份文档（目前是骨架）
3. 第一份 ADR-0001 已起草,请 review
4. 今后原始输入扔 inbox/,别再放根目录
```

---

## 常见陷阱与对策

| 陷阱 | 对策 |
|---|---|
| 用户急着动手,AI 跳过诊断 | 强制 Step 1 不可省,哪怕 30 秒的扫描 |
| 批次 2 用户回答"你看着办" | 把具体选项 ABC 列给他,不让他放权后 AI 乱决定 |
| 迁移后用户发现漏掉文件 | 保留 git commit / zip 快照,随时可恢复 |
| 用户对某个迁移不满 | 单独回滚该条,不影响其他 |
| 重构后结构依然乱 | 做二次检查,该再切分的再切,该合并的合并 |

---

## 极简版（给简单场景用）

如果用户的已有项目很小（<20 文件 + 简单内容）,可以跳过完整三步,用精简版：

```
1. AI 扫一眼,给出 3 条"我打算这么动"的简述
2. 用户点头就动手
3. 完成后简短对照
```

不是所有项目都值得完整的 Retrofit 流程。判断基准：**文件 > 30 个 or 目录层级 > 2 层**才走完整流程。
